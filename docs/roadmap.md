# Delivery Phases

## Phase 1: pgvector-first baseline

- streamable HTTP + legacy SSE
- OAuth/Teleport auth gateway
- knowledge save/get/search/delete
- metadata catalog + in-memory pgvector adapter
- web admin and integration API

## Phase 2: Qdrant adapter

- switchable `vector_backend.active=qdrant`
- parity tests between pgvector and qdrant behavior

## Phase 3: hardening and observability

- structured logs and request IDs
- metrics for auth/search/ingest/session
- retries and circuit-breakers for external LLM APIs
- stricter JWT validation and JWKS caching
- load and soak tests

---

# Планы развития

Пять направлений, расписанные по задачам. Порядок фаз отражает зависимости: административные MCP-инструменты (Phase 4) и новые типы хранилищ (Phase 5) — фундамент; агент (Phase 7) опирается на них; пользовательский интерфейс (Phase 8) — на агента. SKILLS.md (Phase 6) можно вести параллельно с любой фазой.

## Phase 4: управление токенами и пользователями через MCP-инструменты

**Цель:** администрировать SynaMCPs (пользователи, группы, токены, ACL, хранилища) можно из любого MCP-клиента (Claude Code, Cursor), без веб-интерфейса. Это же — основа для автоматизации администрирования агентами (Phase 6, Phase 7).

### 4.1. Модель прав для административных инструментов
- Определить, какие инструменты кому видны: `platform_admin` — всё; `storage_owner`/`storage_admin` — токены и ACL своих хранилищ; обычный пользователь — только свои токены.
- Расширить `evaluateStorageAccess` / access-слой проверками административных операций через MCP-канал (permission-ы `token.policy.manage`, `acl.manage` уже есть в `internal/models/models.go` — переиспользовать, не дублировать).
- Запретить административные инструменты при аутентификации access-токеном (токен сужает права и не должен давать управление токенами) — только JWT/web-принципал.

### 4.2. Набор инструментов
- Токены: `admin_token_create`, `admin_token_list`, `admin_token_get`, `admin_token_revoke`, `admin_token_update_scopes`, `admin_token_update_rate_limit`.
- Пользователи и группы: `admin_user_list`, `admin_user_get`, `admin_user_disable`, `admin_group_list`, `admin_group_members`, `admin_group_add_member`, `admin_group_remove_member`.
- ACL и хранилища: `admin_acl_list`, `admin_acl_grant`, `admin_acl_revoke`, `admin_storage_create`, `admin_storage_archive`.
- MCP-прокси: `admin_mcp_server_list`, `admin_mcp_server_test`, `admin_mcp_scope_set`.
- Для каждого инструмента: JSON-схема входа, типизированный ответ, маппинг на существующие методы `access.Store` / `mcpproxy.Store` (новой бизнес-логики минимум — в основном адаптер MCP → сервисы).

### 4.3. Интеграция в MCP-сервер
- Динамический `tools/list`: административные инструменты появляются только у принципалов с соответствующими правами (по образцу текущей фильтрации knowledge-инструментов в `internal/mcp/server.go`).
- Секрет токена возвращается один раз при создании (как в REST) — зафиксировать это в описании инструмента, чтобы LLM-клиент предупреждал пользователя.
- Аудит: каждая административная операция пишет `AuditEvent` (модель уже есть) с каналом `mcp`.

### 4.4. Тесты и документация
- Табличные тесты видимости инструментов по ролям; негативные тесты (обычный пользователь вызывает `admin_user_disable`, доступ по access-токену).
- E2E-тест: подключение MCP-клиентом, создание токена, отзыв.
- Раздел в `docs/mcp-connection.md` + примеры в SKILLS.md (Phase 6).

**DoD:** полный жизненный цикл токена (создать → сузить → отозвать) выполняется из Claude Code без открытия веб-интерфейса; операции видны в аудите; права проверены тестами.

## Phase 5: новые типы хранилищ — версионированное S3 и Git поверх S3

**Цель:** помимо `knowledge` появляются `StorageKind`: `versioned` (S3 с историей версий документов) и `git` (полноценный git-remote поверх S3 — Go-переписывание [awslabs/git-remote-s3](https://github.com/awslabs/git-remote-s3)).

### 5.1. Общая инфраструктура типов хранилищ
- Расширить `models.StorageKind` (`knowledge` | `versioned` | `git`), провести через каталог, API создания хранилища, админ-интерфейс и ACL (существующие роли применимы ко всем типам).
- Ввести интерфейс поведения хранилища (получение/запись/список), чтобы `knowledge.Service` и новые сервисы не ветвились по `kind` в каждой операции.
- Миграция схемы: поле `kind` уже есть в `storages` — проверить дефолты и обратную совместимость.

### 5.2. Версионированное S3-хранилище
- Дизайн-решение: нативное S3 Versioning бакета vs версии на уровне приложения (ключи `.../documents/{docId}/v{N}`). Рекомендация — прикладное версионирование: работает на любом S3-совместимом бэкенде (MinIO), версии видны в каталоге Postgres.
- Каталог: таблица `document_versions` (docId, version, s3Key, hash, author, createdAt, comment); запись новой версии при каждом обновлении документа.
- API: `GET /api/knowledge/{id}/versions`, `GET /api/knowledge/{id}?version=N`, `POST /api/knowledge/{id}/restore` (восстановление = новая версия).
- MCP-инструменты: `knowledge_versions`, `knowledge_get_version`, `knowledge_restore`.
- Политика хранения: лимит числа версий / TTL на хранилище, фоновая очистка.
- Индексация: в векторном поиске участвует только актуальная версия (перезапись эмбеддингов при обновлении, см. задачу 4.3 плана техдолга).

### 5.3. Git-хранилище поверх S3 — Go-порт git-remote-s3
Референс `awslabs/git-remote-s3` — Python git-remote-helper: git общается с ним по stdin/stdout-протоколу (`capabilities` / `list` / `push` / `fetch`), а состояние репозитория хранится в S3 в виде git-bundle на каждый ref (`<prefix>/<ref>/<sha>.bundle`), атомарность — через листинг и правила разрешения конфликтов. Порт на Go:

- **5.3.1. Пакет `internal/gitremote`** — протокол git-remote-helper: чтение команд из stdin, ответы в stdout (`capabilities`, `list`, `list for-push`, `push`, `fetch`); за основу взять поведение референса, зафиксировать протокольные тесты на golden-сценариях.
- **5.3.2. Слой bundle**: создание бандла (`git bundle create` через exec либо go-git), загрузка/выгрузка из S3 через существующий `internal/storage/blob`; layout ключей — **совместимый с awslabs/git-remote-s3**, чтобы репозитории читались обоими инструментами.
- **5.3.3. Семантика push**: защита от неатомарности — перед записью нового бандла листинг текущих `<sha>.bundle` данного ref; обнаружение конкурентного пуша (несколько head-бандлов) и разрешение как в референсе; удаление устаревших бандлов после успешной записи; force-push и удаление ref.
- **5.3.4. CLI-бинарь `cmd/git-remote-synamcps`** (или `git-remote-s3`-совместимый): схема URL `synamcps://<storage-slug>`, аутентификация access-токеном SynaMCPs (переменная окружения / git credential helper), маппинг storage → бакет/префикс через API сервера.
- **5.3.5. Серверная интеграция**: хранилище типа `git` = префикс в S3 + ACL SynaMCPs; проверка прав на push/fetch по режимам `read`/`read_write` токена; события usage/audit на пуши.
- **5.3.6. Опционально (отдельная итерация): LFS.** Референс включает LFS-агент; в первой версии — явно объявить неподдерживаемым, спроектировать позже.
- **5.3.7. Тесты**: интеграционные против MinIO (push → clone → verify), тест кросс-совместимости с Python-версией (репозиторий, записанный git-remote-s3, читается Go-портом и наоборот), конкурентные пуши.
- Лицензия: awslabs/git-remote-s3 — Apache-2.0; порт с сохранением атрибуции, зафиксировать в NOTICE.

**DoD:** `git clone synamcps://my-repo` и `git push` работают с аутентификацией токеном SynaMCPs; layout в S3 совместим с референсом; версионированное хранилище отдаёт историю и восстанавливает версии через API и MCP.

## Phase 6: SKILLS.md — навык работы с SynaMCPs

**Цель:** файл SKILLS.md в корне репозитория, который агентные инструменты (Claude Code и совместимые) подхватывают как навык: как настраивать SynaMCPs, администрировать его и писать скрипты автоматизации.

### 6.1. Структура и содержание
- Frontmatter (name, description) + разделы:
  - **Установка и настройка**: docker-compose и локальный запуск, структура `configs/*.yaml`, обязательные переменные окружения, настройка провайдеров аутентификации (ссылки на `docs/auth-setup.md`).
  - **Администрирование**: создание хранилищ, выдача ACL, выпуск и сужение токенов, подключение внешних MCP-серверов, диагностика (`/healthz`, `/readyz`, `/metrics`, `/api/admin/status`).
  - **Работа с знаниями**: save/search/get/delete через REST и MCP, ингест файлов и ссылок, видимости personal/group/public.
  - **Автоматизация**: готовые рецепты скриптов — bash + curl против REST API и через MCP-инструменты Phase 4 (ротация токенов, аудит неиспользуемых токенов, массовая выдача ACL, экспорт usage-метрик); соглашения об обработке ошибок и идемпотентности.
- Каждый рецепт — самодостаточный: что делает, какие права нужны, полный пример команды, ожидаемый ответ.

### 6.2. Поддержание актуальности
- Скрипты-примеры вынести в `scripts/examples/` и ссылаться из SKILLS.md; в CI — smoke-прогон примеров против docker-compose окружения.
- Правило: PR, меняющий публичный API (`docs/openapi.yaml`) или набор MCP-инструментов, обновляет SKILLS.md (чек-лист в шаблоне PR).

**DoD:** Claude Code с подключённым SKILLS.md выполняет типовые админ-задачи (выпустить токен на чтение одного хранилища, продиагностировать недоступность поиска) без обращения к остальной документации; примеры проходят в CI.

## Phase 7: чат-агент с памятью на SynaMCPs

**Цель:** встроенный агент-чат, который использует хранилища SynaMCPs как память (поиск и сохранение знаний), и которому в интерфейсе можно назначить набор данных (хранилищ) для работы.

### 7.1. Backend: сервис агента
- Пакет `internal/agent`: цикл работы с LLM (вызов модели → tool calls → результаты → ответ), провайдер LLM конфигурируемый (расширение `internal/llm`: chat-completions с поддержкой tool use, стриминг).
- Инструменты агента = существующие сервисы: `knowledge_search`, `knowledge_get`, `knowledge_save` (+ инструменты прокси-MCP-серверов, доступные принципалу). Все вызовы идут через `access.Service` от имени пользователя чата — агент не расширяет права.
- Привязка набора данных: сущность `AgentDataset` (список storageId, закреплённый за диалогом/агентом); поиск агента скоупится этим списком, пересечённым с реальными правами пользователя.

### 7.2. API чата
- `POST /api/agent/conversations` — создать диалог (с выбором datasets), `POST /api/agent/conversations/{id}/messages` — сообщение, ответ стримится (SSE), `GET .../messages` — история.
- Персистентность диалогов: Postgres (таблицы `agent_conversations`, `agent_messages`), TTL/архивация по конфигу.
- Rate limiting и usage-события на LLM-вызовы (расход токенов LLM в `UsageEvent` — новые поля метрик).

### 7.3. Память агента
- Автосохранение: агент может по запросу пользователя («запомни это») сохранять знания в выбранное хранилище с visibility по умолчанию `personal`.
- Контекст: перед ответом — поиск по датасетам, вставка релевантных фрагментов в промпт (RAG); лимиты объёма контекста в конфиге.
- Прозрачность: в ответе агента — ссылки на использованные документы (docId → ссылка в UI).

### 7.4. Конфигурация и безопасность
- Раздел конфига `agent:` — провайдер/модель, системный промпт, лимиты (сообщений в диалоге, токенов на ответ), список разрешённых инструментов.
- Изоляция: промпт-инъекция из содержимого документов не должна давать агенту операций, недоступных пользователю (гарантируется тем, что все инструменты проходят access-слой); деструктивные операции (`knowledge_delete`) — только с подтверждением в UI.
- Тесты: unit на цикл tool-use (мок-LLM), интеграционные на скоупинг датасетов, негативный тест на попытку выхода за права.

**DoD:** пользователь создаёт диалог, выбирает 2 хранилища, задаёт вопрос — агент отвечает по содержимому именно этих хранилищ со ссылками на документы и может сохранить новое знание; всё в пределах ACL пользователя.

## Phase 8: два варианта интерфейса — админский и пользовательский

**Цель:** текущий веб-интерфейс остаётся интерфейсом администратора; появляется пользовательский интерфейс, в центре которого — чат с агентом (Phase 7).

### 8.1. Разделение и маршрутизация
- Маршруты: `/admin/*` — текущий интерфейс (перенос с `/app`), `/app/*` — новый пользовательский; `/login` общий, после входа редирект по роли (platform_admin → выбор, остальные → пользовательский).
- Ревизия `internal/web/handler.go` и встроенных шаблонов: выделить общий каркас (аутентификация, CSRF, статика) и два независимых бандла.
- Доступ к админ-маршрутам — только с админ-правами (сейчас проверить: серверная проверка, не только скрытие в UI).

### 8.2. Пользовательский интерфейс
- Главный экран — чат с агентом: стриминг ответов (SSE из 7.2), история диалогов, markdown-рендеринг.
- Панель данных: выбор хранилищ-датасетов для агента (только доступные пользователю), просмотр документов, ручной поиск по знаниям.
- Самообслуживание: собственные токены (создать/отозвать — на API Phase 4), настройки подключения MCP-клиентов (существующие connect-конфиги для Cursor/Claude).
- Технология — по текущему подходу (embedded-шаблоны + vanilla JS); если объём чата этого потребует, решение о лёгком фреймворке принять отдельным ADR до начала вёрстки.

### 8.3. Совместимость и выкладка
- Feature flag `web.user_ui.enabled` — выкатка пользовательского интерфейса независимо от админского.
- E2E-смоук на оба интерфейса (вход, чат-сообщение, создание токена в админке).
- Обновить README и `docs/setup.md`: два URL, роли, скриншоты.

**DoD:** администратор и обычный пользователь после логина попадают в разные интерфейсы; пользователь ведёт диалог с агентом и управляет своими датасетами и токенами, не имея доступа к админ-функциям.

## Зависимости между фазами

- Phase 4 (MCP-администрирование) не зависит от остальных; его API переиспользуют Phase 6 (рецепты) и Phase 8 (самообслуживание токенов).
- Phase 5 независима; git-хранилище (5.3) — самая большая отдельная работа, можно вести параллельным треком.
- Phase 7 требует стабильного access-слоя (рекомендуется после Этапов 1–2 плана `docs/architecture-remediation-plan.md`) и желательно Phase 4 для админ-сценариев агента.
- Phase 8 требует Phase 7.
